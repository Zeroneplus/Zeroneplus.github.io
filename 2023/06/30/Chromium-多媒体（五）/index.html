这篇文章继续介绍 EME 相关的内容

在上一篇文章的最后创建了 WebContentDecryptionModuleAccessImpl

在 MediaKeySystemAccessInitializer::RequestSucceeded 中用 WebContentDecryptionModuleAccess 创建了 MediaKeySystemAccess

MediaKeySystemAccess::createMediaKeys

接下来就是使用 MediaKeySystemAccess::createMediaKeys 来创建 MediaKeys

MediaKeySystemAccess::createMediaKeys 内部调用 WebContentDecryptionModuleAccessImpl::CreateContentDecryptionModule

而 WebContentDecryptionModuleAccessImpl::CreateContentDecryptionModule 内部又调用了 CreateCdm 函数

CreateCdm 函数内部调用WebEncryptedMediaClientImpl::CreateCdm，注意 WebContentDecryptionModuleAccessImpl 中有 WebEncryptedMediaClientImpl 的 WeakPtr

WebContentDecryptionModuleImpl::Create

WebContentDecryptionModuleImpl::Create 内部通过 CdmSessionAdapter 去创建 CDM

CdmSessionAdapter 内部使用 cdm_factory 创建 CDM

CdmFactory 是在 MediaFactory::GetCdmFactory 中创建的，在该函数中可以配置使用哪个 CdmFactory

比如可以使用 MojoCdmFactory

MojoCdmFactory 中持有 media::mojom::InterfaceFactory 指针，也就是 MediaInterfaceFactory 的指针：

MojoCdmFactory::Create 函数内部则是通过内部 interface_factory_，也就是 MediaInterfaceFactory 的 CreateCdm 方法

MediaInterfaceFactory::CreateCdm 函数使用自己内部对象 media_interface_factory_ 的 CreateCdm 方法，而 media_interface_factory_ 是 mojo::Remote<media::mojom::InterfaceFactory>，它的另一端是 MediaInterfaceProxy，而 MediaInterfaceProxy 中有 MediaInterfaceFactoryHolder 类的对象，MediaInterfaceFactoryHolder 中有 mojo::Remote<media::mojom::InterfaceFactory> 对象，该对象的另一端是 InterfaceFactoryImpl，所以 MediaInterfaceFactory::CreateCdm 会调用 MediaInterfaceProxy::CreateCdm，然后调用 InterfaceFactoryImpl::CreateCdm

MediaInterfaceFactory::CreateCdm

MediaInterfaceProxy::CreateCdm 内部使用 media_interface_factory_ptr_ 的 CreateCdm 方法，media_interface_factory_ptr_ 是 std::unique_ptr<MediaInterfaceFactoryHolder>

InterfaceFactoryImpl::CreateCdm

最后会调用到 InterfaceFactoryImpl 中

InterfaceFactoryImpl::GetCdmFactory

InterfaceFactoryImpl::CreateCdm 先 GetCdmFactory

GetCdmFactory 通过 mojo_media_client_，而 mojo_media_client_ 是在 MediaService 中

从下面的 GetMediaService 可以看到，如果允许在 GPU 中使用 mojo media，那么会使用 GpuProcessHost::RunService 来得到 MediaService 的 Remote，如果是在 Browser 中使用 mojo media，那么直接使用 media::CreateMediaService 函数创建 MediaService

Browser mojo media

先看 Browser mojo media 的情况，这种情况简单一些

从下面的代码可知，MojoMediaClient 是 AndroidMojoMediaClient 或者 MediaFoundationMojoMediaClient

GPU mojo media

再看 GPU mojo media。

我们知道，Chromium 有 Browser 进程、Render 进程、还有 GPU 进程，Browser 进程只有一个，而每个页面一般会对应一个 Render 进程，而 GPU 进程则是专门负责和 GPU 打交道

在 GPU mojo media 的情况下，GetMediaService 使用 GpuProcessHost::RunServiceImpl 来连接 GPU 进程中的 MediaService

而 GPU 进程中的 MediaService 是在 GpuServiceFactory::RunMediaService 中创建的

上面函数调用了 CreateGpuMediaService 函数，CreateGpuMediaService 函数创建了一个 GpuMojoMediaClient 对象给 MediaService，GpuMojoMediaClient 的方法（如 CreateCdmFactory）会调用一些全局函数（如 CreatePlatformCdmFactory），而这些全局函数在不同平台下有不同的实现，一般会通过编译选项新建相关全局函数的实现，来提供自己平台的 CreateCdm 方法

以 Android 为例，他的 GpuMojoMediaClient 的 CreateCdmFactory 方法调用下面的 CreatePlatformCdmFactory

MojoCdmService::Initialize

在得到 MojoMediaClient 后，会通过他来创建 CdmFactory，然后用该 CdmFactory 来初始化 MojoCdmService

在 Initialize 中通过 cdm_factory 的 Create 方法创建，以 AndroidCdmFactory::Create 为例，下面的函数最后调用了 MediaDrmBridgeFactory 来创建 Android 的 Cdm

结果回传

接下来再看结果回传过程。结果回传主要是通过 cdm_created_cb 回调回传的，把上面的代码从后往前看一遍基本上就清楚了。

MojoCdmService::OnCdmCreated

我们从 MojoCdmService::OnCdmCreated 开始，MojoCdmService::OnCdmCreated 接受 ContentDecryptionModule 的 scoped_refptr 参数，该参数就是 AndroidCdmFactory::Create 中创建的对象

MojoCdmService::OnCdmCreated 中还可能会创建 MojoDecryptorService

InterfaceFactoryImpl::OnCdmServiceInitialized

OnCdmCreated

一路往上，就到了 OnCdmCreated，这里创建了 MojoCdm

再往上，就到了 CdmSessionAdapter::OnCdmCreated，在这里创建了 WebContentDecryptionModuleImpl，注意这里把 CdmSessionAdapter 传给了 WebContentDecryptionModuleImpl

CompleteWebContentDecryptionModuleResult

再往上就是 CompleteWebContentDecryptionModuleResult

WebContentDecryptionModuleResult::CompleteWithContentDecryptionModule 调用内部 impl_ 的 CompleteWithContentDecryptionModule

在 CompleteWithContentDecryptionModule 中用 WebContentDecryptionModule 创建 MediaKeys 并 resolve promise

这一篇文章就到这里，主要介绍了通过 MediaKeySystemAccess 创建 MediaKeys 的过程。
