Chromium base 库之 thread（一）
2023-06-08 22:50:06

之前研究过 Chromium thread 相关内容，打算写几篇文章介绍一下这块的内容。

这篇文章先通过一个例子介绍基本的类结构关系。

示例代码

先看下面的例子（来自 https://keyou.github.io/blog/2019/06/11/Chromium-MessageLoop-and-TaskScheduler/）

上面的代码把 main thread 变成了一个 RunLoop，main thread 会不停的从任务队列中取任务，但是由于上面只有一个线程，所以当 run_loop.Run(); 执行后，main thread 会在执行完 Hello 函数就进入睡眠状态。

实际开发中一般不会这么写，更多的是直接启动一个 thread，然后往该 thread 中 PostTask，但是我们可以以这个例子为研究对象来分析 Chromium thread 的原理。

SingleThreadTaskExecutor

第一步，先创建一个 SingleThreadTaskExecutor 类型的对象，这个类里面有一个非常重要的类型，SequenceManager，的对象

当默认构造 SingleThreadTaskExecutor 时，type 参数为 MessagePumpType::DEFAULT

进而调用下面的构造函数

在构造函数里面会：

  • 创建 SequenceManager
  • 然后从 SequenceManager 创建 TaskQueue
  • 从 TaskQueue 中取得 task_runner，用于后续 post task
  • 最后对 SequenceManager 进行初始化

创建 SequenceManager

SingleThreadTaskExecutor()中通过 sequence_manager::CreateUnboundSequenceManager 这个函数创建 SequenceManagerImpl，而 SequenceManagerImpl 继承 SequenceManager。

SequenceManager::Settings 就是一些参数设置，可以暂时不管

创建 ThreadControllerWithMessagePumpImpl

CreateUnbound 里面先创建 ThreadControllerWithMessagePumpImpl，然后用他来创建 SequenceManagerImpl

ThreadController 顾名思义就是和线程管理相关。

ThreadController 内部持有 MessagePump，而 MessagePump 会反过来调用 ThreadController，ThreadController 则从 SequenceManager 中取 task，调用流程是 RunLoop->ThreadController->MessagePump->ThreadController->SequenceManager。

associated_thread_ 是 scoped_refptr<AssociatedThreadId> associated_thread_;。

AssociatedThreadId 类中有一个成员 std::atomic<PlatformThreadId> thread_id_{kInvalidThreadId};

AssociatedThreadId::CreateUnbound() 函数返回的 AssociatedThreadId 里面 thread_id_ 是 kInvalidThreadId，也就是无效值

继续创建 SequenceManager

上面创建 SequenceManagerImpl 中 controller->GetAssociatedThread() 返回的就是上面 ThreadControllerWithMessagePumpImpl 中的 associated_thread_。

SequenceManagerImpl 会持有 ThreadControllerWithMessagePumpImpl。

注意 controller_->SetSequencedTaskSource(this); 这一步会将 SequenceManagerImpl 指针设置到 ThreadControllerWithMessagePumpImpl 中，后面 ThreadControllerWithMessagePumpImpl 就能从 SequenceManagerImpl 中取任务。

SequenceManagerImpl 中有一个 main_thread_only 的数据结构，main_thread_only 的意思是他只能被执行 RunLoop::Run()的线程访问。thread 模型是这样的：thread 的线程执行 RunLoop.run()，不停的执行任务队列中的 task，其他线程则往 thread 中 post task。

接下来简要分析 SequenceManagerImpl 中重要的数据。

首先是 MainThreadOnly，SequenceManagerImpl 中有 MainThreadOnly 类型的对象，MainThreadOnly 的含义上面已经说了

TaskQueueSelector 用于选择 TaskQueue，一个 SequenceManager 可以管理多个 TaskQueue，当 SequenceManager 希望执行任务时，需要决定从哪个 TaskQueue 中取任务

WakeUpQueue 用于 TaskQueue 给持有 SequenceManager 的线程设置唤醒时间，当 TaskQueue 发送一个 dealyed task 时可能会往 WakeUpQueue 发送一个唤醒时间，指示 main thread 在指定时间后唤醒

SequenceManagerImpl 中持有 ThreadController，但是调用顺序却是 ThreadController 调用 SequenceManagerImpl，后面会看到 ThreadController 会被保存在 thread local storage 中被 RunLoop 类取得。

从 SequenceManager 创建 TaskQueue

先通过 SequenceManagerImpl::CreateTaskQueueImpl 创建 TaskQueueImpl，然后通过 TaskQueueImpl 创建 TaskQueue。TaskQueueImpl 和 TaskQueue 不是继承关系，TaskQueue 类里面有一个成员 std::unique_ptr<internal::TaskQueueImpl> impl_;

TaskQueue::Spec 类如下，暂时不用管

创建 TaskQueueImpl

主线程通过 SequenceManagerImpl 管理 TaskQueueImpl，其他线程通过 TaskQueueImpl 往主线程 post task。

创建好 TaskQueueImpl 后会把他加到 SequenceManagerImpl 的 main_thread_only().active_queues 和 main_thread_only().selector，后面 SequenceManagerImpl 取 task 时会通过 main_thread_only().selector 从 TaskQueueImpl 中取任务。

TaskQueueImpl 有以下重要数据结构

在 MainThreadOnly 中的

  • delayed_work_queue
  • immediate_work_queue
  • delayed_incoming_queue

在 AnyThread 中的

  • immediate_incoming_queue

其他线程往 TaskQueueImpl 中 post task 时，只能送到 AnyThread 的 immediate_incoming_queue，送完之后会根据 task 类型，比如是否 delay，来通知 main thread 处理这个 task，注意这里不是执行 task，而是把 immediate_incoming_queue 中刚刚到送入的 task 移到 MainThreadOnly 中的 queue，执行 task 则有其他的逻辑

从 TaskQueueImpl 构造函数可知，TaskQueueImpl 中有 SequenceManagerImpl 指针，SequenceManagerImpl 中的 WakeUpQueue，还把 SequenceManagerImpl 中的 empty_queues_to_reload_ 设到了 empty_queues_to_reload_handle_（这个和 main thread 唤醒有关，后面再分析）

创建 TaskQueue

创建 TaskQueueImpl 后就可以创建 TaskQueue 了

TaskQueue 的构造函数还创建了 default_task_runner_，default_task_runner_ 是其他线程和 main thread 之间通信的桥梁，其他线程通过 default_task_runner_ 往 main thread 中 post task。default_task_runner_ 是对 TaskQueueImpl 中的 task_poster_ 的包装

TaskQueue 的 task_runner()方法返回的就是上面创建的 default_task_runner_

对 SequenceManager 进行初始化

SetDefaultTaskRunner

SetDefaultTaskRunner 就是设置 main thread 的 TaskRunner，这个函数应该在 main thread 中调用，因为会使用 thread local storage

其实就是把 task_runner 设给 ThreadControllerWithMessagePumpImpl，InitializeThreadTaskRunnerHandle()函数这里不会调用，因为此时 associated_thread_ 还没有进行 bind，InitializeThreadTaskRunnerHandle 会在后面执行

BindToMessagePump

这里是将 MessagePump 绑定到 SequenceManager。

这里同样要求在 main thread 中执行。

之前说过，RunLoop 在调用中间通过 MessagePump 从 SequenceManager 中取任务，MessagePump 后面介绍 RunLoop 时再说

下面函数会将 associated_thread_ 进行绑定，然后将 message_pump 移动到 ThreadController 中，此后 ThreadController 持有 message_pump，RunLoop 会先调用到 ThreadController，ThreadController 再调用到 message_pump，message_pump 又调用 ThreadController，ThreadController 再调用 SequenceManager。

注意上面还通过 RunLoop::RegisterDelegateForCurrentThread 将当前 ThreadControllerWithMessagePumpImpl 传到一个 thread_local 的指针中。

后面创建 RunLoop 实例时会获得这个指针。

在 BindToMessagePump 的最后，由于 task_runner_ 之前在 ThreadControllerWithMessagePumpImpl::SetDefaultTaskRunner 中已经被设置了，所以会执行 InitializeThreadTaskRunnerHandle()。

InitializeThreadTaskRunnerHandle()中会创建 ThreadTaskRunnerHandle，而 ThreadTaskRunnerHandle 的构造函数中会将自己保存在一个 thread_local 的变量中，将来可以在 main thread 中通过 ThreadTaskRunnerHandle::Get()获得该 ThreadTaskRunnerHandle

CompleteInitializationOnBoundThread

这里将 SequenceManagerImpl 设在一个 thread local storage 中。所以 ThreadControllerWithMessagePumpImpl 和 SequenceManagerImpl 都被保存在 thread local storage 中

这篇文章重点介绍了 SequenceManagerImpl、ThreadControllerWithMessagePumpImpl、TaskQueueImpl 的创建以及相互之间的持有关系，下篇文章分析往 TaskQueueImpl 中 post task 的过程。
