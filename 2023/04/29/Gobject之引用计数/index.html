Gobject之引用计数
2023-04-29 19:49:05 # gobject

这篇博客介绍一下 Gobject 的引用计数。

要让 Gobject 类支持引用计数，需要让类继承 GObject 类，之后通过下面的 api 管理对象：

  • 使用 g_object_new 创建对象
  • 使用 g_object_ref 增加引用计数
  • 使用 g_object_unref 减少引用计数

g_object_new

在 g_object_new 时会调用 g_type_create_instance 创建并初始化对象

在 g_type_create_instance 中可以清楚的看到：

  • 先分配内存（包括私有结构）
  • 然后从上往下调用 instance_init

g_object_init

而 GObject 的 instance_init 函数 g_object_init 会初始化引用计数为 1

根据 instance_init 的调用时机，每个 GObject 或其子类的实例在创建时都会执行 g_object_init 函数，所以继承 GObject 的类在创建实例时就会初始化引用计数。

g_object_ref

再看看 g_object_ref 函数，该函数主要是将 ref_count 加 1

g_object_unref

g_object_unref 则是释放引用，同时可能会执行 dispose 和 finalize 函数。

当引用计数大于 1，表示当前可能即便 unref，引用计数也不为 0，此时的逻辑比较简单，尝试进行 CAS 操作来竞争并减小引用计数，如果竞争成功就返回（因为此时肯定不是最后一个 unref 的），竞争失败就重试，此时有可能是最后一个引用

如果重试的时候发现自己的引用计数为 1，那么表示当前是最后一个引用，那么有可能需要析构。此时情况比较复杂，因为可能在析构的时候其他地方发生了 ref 操作。

检查 weak_ref 是否进行了 promote，weak_ref 提升之后引用计数会增加

如果当前引用计数为 1，那么会进行 dispose 操作

dispose 后再判断引用计数，因为在 dispose 时有可能又在其他地方进行了 ref

但是感觉不太应该，因为没有 weak_ref 的话，如果当前自己是最后一个引用，那么其他地方是不可能提升引用的，因为其他地方已经没有指针可用了

如果 ref 大于 1，则尝试 CAS 操作来竞争并减小引用计数，和上面一样，竞争成功则返回，竞争失败就在这里重试。重试之后发现自己就是最后一个引用，那么减小引用计数，减小引用计数之后再判断一下引用计数，因为可能此时其他地方又发生了 ref（这里也不应该，理由同上，在没有 weak_ref 的情况下其他地方不应该增加引用）

如果上一步获取引用计数发现其他地方进行了 ref，就返回，否则执行 finalize 和 g_type_free_instance

有点不理解上面的函数为什么这么设计，既然进入了 dispose 分支，那么当前就是最后一个引用，而 weakref 也是在第 17 行后立马就判断了的，后面代码逻辑似乎是不必要的，如果 weakref 成功 promote，那么当前不应该析构，直接返回，如果 weakref 和当前操作竞争失败，weakref 的 promote 应该返回空，当前操作直接 dispose+finalize。这里的思路和 C++ 标准库的 shared_ptr/weak_ptr 不太一样。

dispose/finalize 的链式调用

关于析构还有一点要注意，我们定义的 dispose、finalize 函数需要链式调用父类的 dispose、finalize 函数，这是因为当前类继承了父类，在初始化时依次调用了父类的 instance_init 函数，所以也需要调用每个父类的 dispose、finalize 函数来释放可能的资源。
