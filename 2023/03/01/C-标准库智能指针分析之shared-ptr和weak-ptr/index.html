C++标准库之shared_ptr和weak_ptr
2023-03-01 20:28:04 # C/C++ # C++标准库

这篇文章介绍一下 shared_ptr 和 weak_ptr。

shared_ptr

shared_ptr 通过引用计数来管理指向的内存，当引用计数为 0 时，释放指向的内存

shared_ptr 内部只有两个指针，__ptr_ 就是我们管理的指针，__cntrl_ 和引用计数有关。

接受裸指针的构造函数

这个构造函数生效的前提是 is_convertible<_Yp*, element_type*> 这个条件要满足，也就是说，_Yp* 不一定要是 element_type* 类型，可以是他的子类

__shared_ptr_default_allocator 定义如下，对于非函数，上面的 _AllocT 就是 allocator<_Yp>，allocator 类还没有仔细看过，不做深入讨论

__shared_ptr_pointer

__cntrl_ 的类型是 __shared_weak_count*，而 __shared_ptr_pointer 是 __shared_weak_count 的子类，__shared_weak_count 又是 __shared_count 的子类，接下来我们从上往下逐个看这些类。

__shared_count

__shared_count 如下，他有一个 __shared_owners_ 成员，代表共享引用计数

增加和减少引用计数都是在这个类里。引用计数最开始创建时为 0，也就是说引用计数是从 0 开始的，真正的引用计数需要在 __shared_owners_ 基础上加 1。

通过 __release_shared() 释放引用计数，如果当前已经没有引用计数了，就会调用 __on_zero_shared()（这个函数的作用是释放真正被管理的内存），并返回 true 或者 false 给调用者。

__on_zero_shared() 是虚函数，留给子类实现。

__shared_weak_count

__shared_weak_count 继承 __shared_count，有一个 __shared_weak_owners_ 成员，就是弱引用计数。

__add_shared() 会调用父类 __shared_count 的 __add_shared()

__add_weak() 就是增加 __shared_weak_owners_

__release_shared() 先调用父类的 __shared_count::__release_shared()，如果返回 true，表示当前是最后一个 shared 计数，则调用 __release_weak() 来减小弱引用计数。这背后的逻辑是：所有的 shared 引用整体要对应一个 weak 引用，因为最开始创建 shared_ptr 时，shared 引用和 weak 引用都被初始化为 0（需要 +1，结果就是 1）。这个逻辑很关键，理解了这一点就理解了 shared_ptr。

在 __release_weak() 中，先尝试用 Acquire 的内存序去读取 __shared_weak_owners_，如果返回 0（引用计数初始值为 0，要加 1），表示当前一定没有其他的 weak_reference（只有自己这一个 weak_reference），因为如果有了其他的 weak_reference，一定会增加引用计数；如果第一步读取的 __shared_weak_owners_ 大于 0，表明还有其他的 weak_reference，这个时候再尝试 compare and swap 操作来减小 __shared_weak_owners_。这其实是一种优化手段，如果 Acquire 读就能判定当前是最后一个 weak_reference，那就不需要再使用 compare and swap 操作，Acquire 读更轻量级一些。

__on_zero_shared_weak() 也是一个纯虚函数需要子类实现，在这个函数里会释放辅助内存。

再看 __shared_weak_count::lock 函数，这个函数被 weak_ptr 里的 lock 函数调用。它先进行 load 读取引用计数，然后尝试 compare and swap 来增加引用计数，不理解这种 cas 操作的可以查查相关资料。

__shared_ptr_pointer

__shared_ptr_pointer 继承自 __shared_weak_count。

__shared_ptr_pointer 存储了指针、删除器、allocator 成员。他还实现了父类的 __on_zero_shared()、__on_zero_shared_weak() 两个纯虚函数

__on_zero_shared() 是共享计数为 0 时调用，该函数会释放真正管理的指针，同时析构删除器

  • __data_.first().first() 就是被管理的指针
  • __data_.first().second() 就是删除器

__on_zero_shared_weak() 是弱引用为 0 时调用，该函数会析构 allocator，然后释放 __shared_ptr_pointer 自己的内存（注意下面 shared_ptr 中有两个指针，都需要 delete）。也就是说其实有两块内存需要释放，一块是被管理的内存，一块是辅助的内存，被管理内存是共享引用为 0 时释放，辅助内存是弱引用计数为 0 时释放

从上面的分析可以发现：智能指针本质就是通过一个生命周期比目标指针长的引用计数对象来管理目标指针。

__shared_ptr_emplace

make_shared 中使用了 __shared_ptr_emplace 类来作为 _CntrlBlk，__shared_ptr_emplace 也继承自 __shared_weak_count，但是 __shared_ptr_emplace 中 __data_ 成员里的模板参数是 _Tp，不是 _Tp*，也就是说被管理的对象会和 __shared_ptr_emplace 一起分配内存。这就是为什么 make_shared 要少分配一次内存。

weak_ptr

在上面的 shared_ptr 中，有强弱两种引用计数，强引用（或者叫共享引用）计数为 0 时释放被管理的内存，弱引用计数为 0 时释放辅助对象的内存。

标准库中除了 shared_ptr，还有 weak_ptr。weak_ptr 看名字就知道是“弱指针”，肯定和弱引用计数有关，我们大概可以推测创建一个 weak_ptr 就会增加弱引用计数，而事实也确实如此

weak_ptr 需要和 shared_ptr 一起使用，用来避免循环引用问题

weak_ptr 的成员数据很简单

一个指向被管理对象的指针 __ptr_，一个指向 __shared_weak_count 的指针 __cntrl_，也就是说 weak_ptr 同样也持有 shared_ptr 中的辅助对象指针

weak_ptr 的构造函数接受一个 shared_ptr 对象

上面构造函数内部就是把 shared_ptr 的辅助对象指针赋值到 weak_ptr 内部的辅助对象指针。所以 weak_ptr 和他关联的 shared_ptr 共享一个辅助对象

在上面的构造函数中还调用了辅助对象的 __add_weak 方法增加弱引用计数。而 weak_ptr 析构时调用辅助对象的 __release_weak() 方法减少弱引用计数

通过上面 shared_ptr 和 weak_ptr 可知，如果使用 make_shared 创建 shared_ptr，然后用创建的 shared_ptr 再去创建 weak_ptr，此时如果 shared_ptr 析构，那么就没有共享引用只有弱引用，但 make_shared 创建的对象内存是在辅助对象中，也就是在这种情况下可以通过 weak_ptr 得到还没被释放内存的被管理对象的指针。但是要注意，通过这种方式得到的对象是调用了析构函数的。

enable_shared_from_this<T> 内部有一个 weak_ptr<T>，通过这个 weak_ptr<T> 就可以创建 shared_ptr<T>。

为什么有 weak_ptr<T> 就可以了，他不是弱引用吗？这是因为 shared_from_this()只能从 shared_ptr 中进行调用，这就保证在对 weak_ptr<T> 进行 lock 时至少有一个共享引用，所以 enable_shared_from_this<T> 内部的 weak_ptr<T> 的 lock 是必定成功的。
