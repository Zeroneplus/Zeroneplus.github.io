ffmpeg 之 muxer/encoder 相关 api (二)
2023-05-05 20:35:21 # ffmpeg

这篇文章接着上一篇继续分析剩下的函数。

avcodec_send_frame

编解码调用的送数据接口：

============================================
encoder: avcodec_send_frame, avcodec_receive_packet
muxer: av_write_frame
============================================
demuxer: av_read_frame
decoder: avcodec_send_packet, avcodec_receive_frame
============================================

先看看 avcodec_send_frame。avci 中有个 buffer_frame，用于缓存输入的 frame，如果 buffer_frame 已经有了数据，那么这次 avcodec_send_frame 不会发送 frame，会返回 AVERROR(EAGAIN)。如果输入的 frame 为空，那么进入 draining 模式，后续不会再接受输入。如果 buffer_pkt 这个输出缓存为空，就尝试调用 encode_receive_packet_internal 进行解码，而 encode_send_frame_internal 其实没有做解码。

encode_send_frame_internal

这个函数就是把输入的 src AVFrame ref 到 avci->buffer_frame

encode_receive_packet_internal

这个函数执行了解码

encode_simple_receive_packet

看更常见的 encode_simple_receive_packet 解码逻辑

encode_simple_internal

这里又出现了 in_frame，之前是把输入 frame 放在了 buffer_frame 中，如果 in_frame 为空，会尝试调用 ff_encode_get_frame 把 buffer_frame 移动到 in_frame。取到 in_frame 后调用 codec 中定义的 encode 函数进行编码

avcodec_receive_packet

这个函数用于从 encoder 中取数据，由于之前 avcodec_send_frame 已经尝试主动解码了，所以这个函数一般可以直接取到 buffer_pkt，如果 buffer_pkt 为空，则调用 encode_receive_packet_internal 进行解码

av_write_frame

这个函数把 packet 写到 muxer 中，为什么不叫 write_packet?

可以看到，下面出现了 parse_pkt，估计也是缓存 buffer

write_packets_common

上面函数内部调用了 write_packets_common 来写入 packet。

check_packet、prepare_input_packet、check_bitstream 都没有做什么重要的事

write_packet_common

write_packet_common 才是真正写 packet 的地方

write_packet

在 write_packet 里面调用 write_uncoded_frame 或者 write_packet
