C++ 标准库之 pair
2023-05-17 22:32:59 # C/C++ # C++标准库

这篇文章介绍一下 pair。

pair 的结构比较简单，有两个成员数据分别代表第一个和第二个成员。pair 没有像 __compressed_pair 一样采用空基类优化策略。

pair 没有 first()、second()函数，但可以直接访问 first、second 成员

默认构造函数

默认构造函数生效的条件是要求 _CheckArgsDep<_Dummy>::__enable_explicit_default() 条件为 true。这在模板编程中很常见，模板函数经常会对参数施加限制，如果参数不满足要求那么这个模板就会被排除。

接下来就看看这个条件

默认条件下 _Dummy 为 true，所以 _MaybeEnable 为 true，那么根据 conditional 的语义，_CheckArgsDep<_Dummy> 就是 _CheckArgs

__enable_explicit_default 要求 _T1 和 _T2 都是可以默认构造，这个很好理解，因为没有输入参数，所以只能默认构造。__enable_explicit_default 还要求 !__enable_implicit_default。__enable_implicit_default 又检查 __is_implicitly_default_constructible

__is_implicitly_default_constructible 检查两个条件：_Tp 要能 default_constructible，以及 _Tp const& tmp = {} 合理的，只有这两个条件都满足 __is_implicitly_default_constructible 才返回 true。

默认构造函数还有一个允许 implicit 转换的版本：

这个模板函数检查的条件是上面说过的 __enable_implicit_default。

可以发现，implicit 的构造和 explicit 的构造是互斥的，二者只能出现一个，因为 __enable_implicit_default 和 __enable_explicit_default 是互斥的。

这种定义 implicit/explicit 两个版本的构造函数在很多标准库中都有出现，至于为什么这样做在下面解释。

接受两个参数的拷贝构造

也有 explicit 和 implicit 的版本

这个函数检查 __enable_explicit 条件

这个条件除了要求 _U1 能以括号初始化 _U1 b; first_type a(b); 构造 first_type，还要求 _U1 不能以=初始化 _U1 b; first_type a = b; 构造 first_type。

而 __enable_implicit 则要求 _U1 能以括号初始化 _U1 b; first_type a(b); 构造 first_type，以及 _U1 能以=初始化 _U1 b; first_type a = b; 构造 first_type。

个人猜测之所以这样设计，是为了保持括号初始化的一致：

当 pair 使用上面的 ={...} 初始化时，如果构造函数是 explicit 的，那么此构造函数是无法访问的。

而上面模板函数根据 is_convertible<_U1, first_type> 和 is_convertible<_U2, second_type> 这两个条件决定是使用 implict 还是 explicit 版本：当 _U1、_U2 允许=初始化时使用 implict，否则时用 explicit。

这背后的逻辑是，如果 pair 内部的元素不支持=初始化，那么 pair 就不应该支持 ={...} 初始化。

参见：https://en.cppreference.com/w/cpp/language/explicit

接受泛型 pair 的构造函数

有拷贝构造和移动构造两个版本。

为什么接受右值的版本，forward 时使用 _U1 为参数？而没有考虑 __p 是左值引用还是右值引用

因为当 _U1 为值类型（比如 int、char）时，forward<_U1> 会得到右值引用，当 _U1 为左值引用类型（比如 int&），forward<_U1> 得到左值引用，刚好符合当前参数 __p 的右值语义：如果 __p 里面是值，那么应该传它的右值引用从而发生移动，如果 __p 里面是引用，那么此时即便是右值语义，也应该传递引用
