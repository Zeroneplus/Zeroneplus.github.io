注意 pthread_t、__libcpp_thread_id、__libcpp_thread_t、__thread_id：__libcpp_thread_id 和 __libcpp_thread_t 都是 pthread_t 的别名，而 __thread_id 是对 __libcpp_thread_id 的包装，他有一个 private 的 implicit 的构造函数（接受 __libcpp_thread_id 参数），从而可以有 __libcpp_thread_id 到 __thread_id 的自动转换，但是该转换只能在友元，比如 this_thread::get_id() 函数中访问

__thread_proxy 中重新使用 unique_ptr 包装输入的 __vp 指针，而 thread 构造函数会对 unique_ptr 进行 release 操作。

其他的操作，lock、try_lock、unlock 和 pthread 是一样的

lock_guard

lock_guard 是 mutex 的辅助类。lock_guard 的构造函数接受一个锁类型的变量，同时在构造函数内加锁，还会在析构时释放锁，所以加锁和 lock_guard 的生命周期是关联的。lock_guard 还有一个构造函数，他可以接受一个已经加锁的锁变量，要使用这个构造函数，第二个参数必须是 adopt_lock_t 类型。

unique_lock

unique_lock 和 lock_guard 类似，都提供了将锁和自己的生命周期关联起来的功能。但是 unique_lock 语义更丰富，有移动构造和移动赋值，还提供了 defer_lock、try_to_lock 两种构造语义，defer_lock 就是构造时不加锁，后面可以自己主动加锁，try_to_lock 则是在构造时使用 try_to_lock 而非 lock。

condition_variable

C++11 的条件变量和 pthread 条件变量语义基本类似，都有 wait、signal 等动作。condition_variable 只接受 unique_lock，而且传给 wait 的 unique_lock 必须上锁

once_flag 和 call_once

C++11 线程库也有 thread_once，但是他没有使用 pthread 的 thread_once，而是自己实现了 thread_once，原理和之前介绍过的 glib 的 thread_once 类似。同样要注意，once_flag 也要求为静态生命周期。

call_once 函数也是先检查 once_flag 是否为 ~0，判断函数是否已经执行完成，如果没有执行，或者正在执行，则调用 __call_once 做进一步判断。

call_once 函数创建的 typedef tuple<_Callable&&, _Args&&...> _Gp; 的里面是引用类型，因为根据模板参数推断原理，_Callable 和 _Args 一定是 T 或者 T&，再在其右边加 &&，折叠之后一定是 T& 或者 T&&。所以 tuple 里面存的是引用类型。

在 __call_once 函数中，通过判断 flag 是否为 1 来确定目标函数是否正在被执行，但是没执行完，如果当前线程发现 flg 为 0，那么表示还没有其他线程执行过目标函数，于是尝试执行目标函数并将 flag 设为 1 来让其他线程等待，最后执行完目标函数后，将 flag 设为 ~0，flag 为 ~0 表示目标函数已经执行完成，其他线程会从等待中退出。
