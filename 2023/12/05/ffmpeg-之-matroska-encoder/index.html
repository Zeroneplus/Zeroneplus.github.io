ffmpeg 之 matroska encoder
2023-12-05 22:14:05 # ffmpeg # mkv

最近在做一个录像功能优化的任务，需要分析 mkv 格式和 mp4 格式的优缺点，于是阅读了 ffmpeg matroska encoder 代码以及 mkv 格式的官方 spec。特此记录一下 mkv 格式的学习心得。

mkv 文件可以理解为一个二进制格式的 xml 文件，常规的 xml 文件如下，通过 < /> 来标识元素

而 mkv 使用 ebml（Extensible Binary Meta Language）格式来存储数据。ebml 中每个元素都有三个部分：element id（全局唯一，用于标识元素类型）、element data length（element data 的长度）、element data。element id 和 element data length 都使用变长整数来存储，变长整数是普通整数的压缩，变长整数分为 3 部分，前缀 0、分隔符（1 比特的 1）、数据。前缀 0 和分隔符用于确定当前变长整数的字节数（前缀 0 的个数 +1），举几个例子：

所以对于变长整数来说，1byte 最多可以使用 7bit，2byte 最多可以使用 14bit，3byte 最多可以使用 21bit，以此类推。变长整数和 h264 格式中的指数哥伦布编码类似，但是变长整数要求 byte-aligned。更多关于 ebml 的内容请参考 https://datatracker.ietf.org/doc/rfc8794/。

把上面 ebml 理解之后，mkv 格式就简单了。mkv 和 mp4 有很多相似的地方，理解的时候可以把 mp4 中的概念迁移到 mkv 中：mkv 的 element 类似 mp4 中的 box，element id、element data length 类似 mp4 box 的 type field、size field；mkv 也存在元素的嵌套，父元素可以包含若干子元素。

这篇文章不打算介绍 mkv spec 相关内容，这些内容如果你理解 mp4 格式，很快就能触类旁通。这篇文章想结合 matroskaenc.c 来分析 mkv 格式某些不容易理解的地方。

codec 相关的 extra data 如何存储

和 mp4 类似，mkv 中有一个 codecpriv element，位于 track->trackentry 下，用于存储 codec 相关的 extra data

mkv_assemble_native_codecprivate 里面也是调用 ff_isom_write_avcc 来得到 h264 格式的 extradata

mkv 格式是如何存储 sample 数据的

数据存在 cluster element 中，mkv 格式原生支持切片，默认情况下 5s 或者 5MB 分一个小片

在写 pkt 的时候，会检查当前积攒的 buffer 是否已经足够（大于 5s 或者 5MB），如果是，就将当前积攒的数据作为一个 cluster 写入到文件中

如果数据还不够，就把 sample 数据缓存，写到 block element 中

上面代码有以下几点要注意

  • 每个 cluster element 的头部有一个 CLUSTERTIMECODE，这个时间再加上 block 里的时间才是 block 中 sample 的 pts，block 里的时间用 2byte 来表示，所以当 block 里的时间大于 int16_t 能表示的极限时就需要分片

  • mkv 格式只存储 pts，不存储 dts（某些 pts=dts 的场景中会存储 dts），所以 mkv 支持 variable fps，因为两个 sample 之间的 duration 由他们的 pts 来决定

  • mkv 也存在关键帧索引的概念，关键帧索引存储在 cues element 中，mkv 只会在以下条件下存储当前 sample 的索引信息：当前sample是关键帧 && (当前sample是video/subtitle || 当前是audio且整个文件没有video且当前sample是当前cluster的第一个sample)。这就避免了对每一个 audio sample 都保存一个关键帧索引。

    • 下面存储 cuepoint 时，cluster_pos 参数代表当前 cluster 在整个文件的起始位置，而 entries[idx].cluster_pos 是 cluster_pos - mkv->segment_offset，所以 entries[idx].cluster_pos 就是当前 cluster 相对 segment element 的偏移值；entries[idx].pts 则是当前关键帧的绝对时间；entries[idx].relative_pos 则是当前关键帧存储位置相对当前 cluster 的偏移

最后整合所有 cuepoint 代码如下

会将相同时间的关键帧信息存储在一个 cuepoint 中。在 cuepoint 的头部存储一个 entry->pts，也就是绝对的 pts，然后存储若干个 CUETRACKPOSITION，每个 CUETRACKPOSITION 包含下述元素

  • track_num 当前属于哪个 track
  • CUECLUSTERPOSITION 当前 sample 所处的 cluster 相对于 segment 的位置（offset 可以忽略，他用于 cues 移动到文件头部时调整 cluster_pos，entry->cluster_pos + offset 本质还是当前 sample 所处的 cluster 相对于 segment 的位置）
  • CUERELATIVEPOSITION 当前 sample 相对于当前 cluster 的偏移

有了上面这些信息，给定一个指定的 pts，就可以 seek 到相应的 cluster 中。
